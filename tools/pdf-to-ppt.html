What comes through

  • Paragraphs as single text boxes, so a paragraph stays one thing you can edit.
  • Tables with ruling lines, rebuilt as real PowerPoint tables with their fills sampled from the page.
  • Fonts mapped to metric-compatible equivalents, with bold and italic kept per run.
  • Backgrounds, charts and vector art, rendered behind the text so nothing vanishes.
  • Point sizes and positions as they were in the PDF, one slide per page.

Where it stops

A browser cannot see which fonts are installed on your PC, so each source font maps to a standard equivalent from a fixed table rather than being verified against your machine. The desktop app does that verification, which is the one difference between the two. Tables drawn with spacing alone and no ruling lines arrive as positioned text rather than as a table, and a scanned page has no text to recover, so it arrives as a picture of itself.

How to convert PDF to PowerPoint

  • Drop your PDF into the box above, or choose a file.
  • Wait while each page is rebuilt as a slide on your own device.
  • Click Download to save the .pptx file.
  • Open it in PowerPoint, Google Slides or Keynote.

What happens to my tables?

A table with ruling lines is detected from those lines and rebuilt as a real PowerPoint table, so you can click a cell, retype it, and add rows. Cell fills are sampled from the page so shading survives. This is checked on every release by a regression suite that fails the build if any table in the test documents comes back as loose text.

Can I edit the text afterwards?

Yes, on any page that had selectable text. Paragraphs become PowerPoint text boxes at paragraph level rather than line level, which matters: a line-level conversion looks similar until you try to edit a sentence and find it split across five boxes that will not reflow.

Does my file get uploaded?

No. The conversion runs as JavaScript inside your own browser: the PDF is read on your device and the .pptx is built there too. You can watch your browser's network activity during a conversion and see that nothing carries your file anywhere.

Is this the same engine as the desktop app?

It is the same pipeline, ported, and both are held to the same regression suite: identical test documents, identical assertions, side by side on every release. The one difference is font handling. The desktop app verifies fonts against the ones installed on your PC, which a browser is not allowed to see, so the browser maps to standard equivalents instead.

What about a scanned PDF?

A scan is a photograph of a page, so there is no text to recover. That page arrives as a picture of itself: nothing is lost visually, but the text is not editable. Recognising the words needs OCR, which BenchPDF does not include.

Is there a page limit?

None is set by the tool. The work happens on your own device, so your machine's memory is the practical ceiling. A short document converts in seconds; a long one with many tables takes noticeably longer, and the desktop app is the more comfortable place for very large files.
